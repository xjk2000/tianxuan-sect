# 两阶段审查

## 技能说明

执法堂对各堂完成的任务进行两阶段审查：先验证功能符合性，再验证代码质量。

## 核心原则

**分离关注点** - 先确保功能正确，再确保质量优秀。

## 阶段划分

### 阶段1: 功能符合性测试

**目标**: 验证是否实现了验收标准

**测试内容**:
- 读取任务总览中的验收标准
- 逐项验证每个验收标准
- 记录测试结果

**通过条件**: 所有验收标准都通过

### 阶段2: 代码质量测试

**目标**: 验证代码质量是否符合标准

**测试内容**:
- 边界测试（空值、边界值、超长、特殊字符）
- 异常测试（网络异常、数据库异常、第三方服务异常）
- 性能测试（响应时间、并发能力、资源占用）
- 安全测试（SQL注入、XSS、敏感信息泄露、权限控制）

**通过条件**: 所有质量测试都通过

## 执行流程

### Phase 1: 功能符合性测试

#### 1.1 读取验收标准

```bash
cat 天玄宗/宗门任务榜/{taskId}/任务总览.md | grep -A 100 "## 验收标准"
```

提取所有验收标准项。

#### 1.2 逐项验证

对每个验收标准：

```
验收标准1: 用户注册成功，返回 userId
测试步骤:
1. 准备测试数据: username=test001, email=test@example.com, password=123456
2. 调用注册接口: POST /api/auth/register
3. 验证返回结果: code=200, data.userId 不为空

预期结果: 返回 userId
实际结果: 返回 userId=12345
状态: ✅ 通过
```

#### 1.3 生成报告

```bash
cat > 天玄宗/宗门任务榜/{taskId}/执法堂-功能符合性报告.md <<EOF
# 功能符合性测试报告

**测试时间**: $(date +"%Y-%m-%d %H:%M")
**测试人**: @执法堂长老

## 测试结果

### 验收标准1: 用户注册成功，返回 userId
- 状态: ✅ 通过
- 测试步骤: [步骤]
- 预期结果: [预期]
- 实际结果: [实际]

### 验收标准2: 参数校验失败返回 400 错误
- 状态: ✅ 通过
- 测试步骤: [步骤]
- 预期结果: [预期]
- 实际结果: [实际]

## 总结

- 通过: 10 项
- 失败: 0 项
- 总计: 10 项

**结论**: ✅ 功能符合性测试通过
EOF
```

#### 1.4 判断是否进入阶段2

- **全部通过** → 进入阶段2
- **有失败** → 进入 Systematic Debugging，打回修复

### Phase 2: 代码质量测试

#### 2.1 边界测试

```
边界测试1: 空值处理
测试内容: 传入空字符串 username=""
预期结果: 返回 400 错误，提示"用户名不能为空"
实际结果: 返回 400 错误，提示"用户名不能为空"
状态: ✅ 通过

边界测试2: 超长字符串
测试内容: 传入 1000 字符的 username
预期结果: 返回 400 错误，提示"用户名长度不能超过50"
实际结果: 返回 400 错误，提示"用户名长度不能超过50"
状态: ✅ 通过

边界测试3: 特殊字符
测试内容: 传入 username="<script>alert(1)</script>"
预期结果: 返回 400 错误或转义处理
实际结果: 返回 400 错误，提示"用户名包含非法字符"
状态: ✅ 通过
```

#### 2.2 异常测试

```
异常测试1: 网络超时
测试内容: 模拟网络超时（设置超时时间为1ms）
预期结果: 返回 500 错误，提示"网络异常"
实际结果: 返回 500 错误，提示"网络异常，请稍后重试"
状态: ✅ 通过

异常测试2: 数据库异常
测试内容: 停止数据库服务
预期结果: 返回 500 错误，提示"系统异常"
实际结果: 返回 500 错误，提示"系统异常，请稍后重试"
状态: ✅ 通过
```

#### 2.3 性能测试

```
性能测试1: 接口响应时间
测试内容: 调用注册接口 100 次，统计平均响应时间
要求: < 2s
实际: 平均 1.2s，最大 1.8s
状态: ✅ 通过

性能测试2: 并发能力
测试内容: 并发 100 个请求
要求: 成功率 > 95%
实际: 成功率 98%
状态: ✅ 通过
```

#### 2.4 安全测试

```
安全测试1: SQL 注入
测试内容: username="admin' OR '1'='1"
预期结果: 不会导致 SQL 注入
实际结果: 返回 400 错误，参数校验失败
状态: ✅ 通过

安全测试2: XSS 攻击
测试内容: username="<script>alert(1)</script>"
预期结果: 不会执行脚本
实际结果: 返回 400 错误或转义处理
状态: ✅ 通过

安全测试3: 敏感信息泄露
测试内容: 检查日志和错误信息
预期结果: 不包含密码、token 等敏感信息
实际结果: 日志中密码已脱敏
状态: ✅ 通过
```

## 测试工具

### 自动化测试

```bash
# 单元测试
npm test
mvn test
pytest

# 集成测试
npm run test:e2e
mvn verify

# 性能测试
ab -n 1000 -c 10 http://localhost:8080/api/users
```

### 手动测试

```bash
# API 测试
curl -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"username":"test001","email":"test@example.com"}'

# 边界测试
curl -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"username":"","email":"test@example.com"}'
```

## 好处

1. **分离关注点** - 先功能后质量，避免混淆
2. **提高效率** - 功能不对就不用测质量
3. **全面覆盖** - 功能和质量都不遗漏
4. **清晰报告** - 两份报告，问题定位清晰

## 注意事项

1. **必须按顺序** - 先阶段1后阶段2
2. **阶段1失败不进入阶段2** - 避免浪费时间
3. **两个阶段都要认真** - 不要偏废
4. **记录详细** - 便于问题定位和修复
