# 宗门新任务

## 概述

`/宗门新任务` 命令用于启动一个新的开发任务。宗主会主导需求探索、任务拆解和工作分配，确保任务目标清晰、计划完善。

## 🚨 重要提醒给宗主

**你必须按照以下完整流程执行，不可跳过任何步骤：**

1. **Phase 1: 需求探索** - 使用需求探索术深入理解需求
2. **Phase 2: 任务拆解** - 使用任务计划编写术创建详细计划（遵循 No Placeholders 原则）
3. **Phase 3: 创建任务目录和文件** - 必须实际创建以下内容：
   ```bash
   # 生成任务ID
   TASK_ID=$(date +"%Y-%m-%d")-[任务简称]
   
   # 创建任务分支
   git checkout -b task/$TASK_ID
   
   # 创建任务目录
   mkdir -p 天玄宗/宗门任务榜/$TASK_ID
   
   # 写入任务文件（必须包含完整内容，不能是占位符）
   # - 任务总览.md
   # - 执行计划.md
   # - 阵堂任务.md（如需要）
   # - 器堂任务.md（如需要）
   # - 执法堂验收.md
   
   # 创建状态文件
   # .task-status.json
   ```
4. **Phase 4: 立即开始执行任务** - 创建任务后，**必须立即**调度各堂开始执行，不要等待用户再次指示

**HARD-GATE 检查点**：
- ❌ 未经用户确认设计方案前，禁止创建任务文件夹
- ✅ 用户确认设计后，必须立即创建完整的任务目录结构
- ✅ 所有任务文件必须包含完整内容，不能有 TODO、TBD 等占位符
- ✅ 创建任务后，必须立即开始执行，调度第一个可执行的长老（通常是藏经阁长老）

## 执行流程

### 第一阶段：需求探索（宗主）

宗主会使用 **需求探索术** 进行深入的需求分析：

1. **理解用户意图**
   - 询问关键问题澄清需求
   - 了解业务背景和使用场景
   - 确认技术约束和性能要求

2. **探索项目上下文**
   - 检查现有代码结构
   - 查看配置文件和依赖
   - 查看 git 历史了解最近变更
   - 查询藏经阁中的灵脉信息

3. **委派藏经阁调研**（如需要）
   - 如涉及新技术或第三方平台
   - 藏经阁长老调研官方文档
   - 将调研结果存入功法大全

4. **设计方案讨论**
   - 提出多个技术方案
   - 分析各方案优劣
   - 与用户确认最终方案

### 第二阶段：任务规划（宗主）

确认方案后，宗主**必须**使用 **任务计划编写术** 创建详细的执行计划并**实际创建任务目录和文件**：

#### 步骤 1: 生成任务 ID 和创建分支

```bash
# 生成任务ID（格式：YYYY-MM-DD-任务简称）
TASK_ID=$(date +"%Y-%m-%d")-user-login  # 示例

# 创建并切换到任务分支
git checkout -b task/$TASK_ID
```

#### 步骤 2: 创建任务目录结构

```bash
# 创建任务目录
mkdir -p 天玄宗/宗门任务榜/$TASK_ID

# 目录结构如下：
# 天玄宗/宗门任务榜/<任务ID>/
# ├── 任务总览.md          # 需求和方案说明
# ├── 执行计划.md          # 详细的分步计划
# ├── 阵堂任务.md          # 后端开发任务（如需要）
# ├── 器堂任务.md          # 前端开发任务（如需要）
# ├── 执法堂验收.md        # 测试验收标准
# └── .task-status.json    # 任务状态跟踪
```

#### 步骤 3: 编写任务文件（必须包含完整内容）

**任务总览.md** - 必须包含：
- 需求描述
- 技术方案
- 数据库设计（如有）
- API 设计（如有）
- 验收标准

**执行计划.md** - 必须包含：
- 完整的实现步骤
- 每步的具体代码
- 测试命令和预期输出
- 遵循 No Placeholders 原则

**阵堂任务.md / 器堂任务.md** - 必须包含：
- 完整的代码实现
- 具体的文件路径
- 详细的测试步骤

**执法堂验收.md** - 必须包含：
- 功能测试用例
- 质量审查标准
- 集成测试步骤

#### 步骤 4: 创建状态文件

```bash
cat > 天玄宗/宗门任务榜/$TASK_ID/.task-status.json <<EOF
{
  "taskId": "$TASK_ID",
  "status": "in_progress",
  "createdAt": "$(date -u +"%Y-%m-%dT%H:%M:%SZ")",
  "createdBy": "@宗主",
  "branch": "task/$TASK_ID",
  "description": "[任务描述]",
  "tasks": {
    "藏经阁": {"status": "pending", "blockedBy": []},
    "阵堂": {"status": "pending", "blockedBy": ["藏经阁"]},
    "器堂": {"status": "pending", "blockedBy": ["阵堂"]},
    "执法堂": {"status": "pending", "blockedBy": ["阵堂", "器堂"]}
  }
}
EOF
```

#### 步骤 5: 提交任务文件

```bash
git add 天玄宗/宗门任务榜/$TASK_ID/
git commit -m "feat: 创建任务 $TASK_ID"
```

### 第三阶段：立即开始执行（宗主）

**重要**：任务创建完成后，宗主**必须立即**开始执行，不要等待用户再次指示。

宗主会按照依赖关系自动调度各堂执行：

**宗主会立即执行以下调度**：

1. **藏经阁长老**（如需要）
   ```
   @藏经阁长老
   
   **任务文件**: /天玄宗/宗门任务榜/$TASK_ID/藏经阁任务.md
   
   请按照任务文件中的要求完成调研工作。
   ```

2. **阵堂长老**（后端开发，等藏经阁完成后）
   ```
   @阵堂长老
   
   **任务文件**: /天玄宗/宗门任务榜/$TASK_ID/阵堂任务.md
   
   请按照任务文件中的详细步骤实现后端功能。
   ```

3. **器堂长老**（前端开发，等阵堂完成后）
   ```
   @器堂长老
   
   **任务文件**: /天玄宗/宗门任务榜/$TASK_ID/器堂任务.md
   
   请按照任务文件中的详细步骤实现前端功能。
   ```

4. **执法堂长老**（测试验收，等阵堂和器堂都完成后）
   ```
   @执法堂长老
   
   **任务文件**: /天玄宗/宗门任务榜/$TASK_ID/执法堂验收.md
   
   请进行两阶段测试验收。
   ```

**执行顺序**：
- 如果需要调研 → 先调度藏经阁长老
- 藏经阁完成后 → 自动调度阵堂长老
- 阵堂完成后 → 自动调度器堂长老（如有前端任务）
- 阵堂和器堂都完成后 → 自动调度执法堂长老

## 使用场景

### 场景一：新功能开发

```
用户: /宗门新任务

用户: 我想实现一个用户登录功能

宗主: 收到！让我先了解一下需求...
      
      **需求探索**：
      1. 使用什么认证方式？JWT 还是 Session？
      2. 需要支持第三方登录吗（微信、GitHub等）？
      3. 密码复杂度要求？
      4. 是否需要记住登录状态？
      5. 登录失败次数限制？

用户: 使用 JWT，暂不需要第三方登录，密码至少 8 位包含字母数字，需要记住登录，失败 5 次锁定 30 分钟

宗主: 明白了！我会：
      
      **技术方案**：
      - 后端：Spring Security + JWT
      - 前端：Vue 3 + Pinia 状态管理
      - 数据库：MySQL 用户表 + Redis 缓存
      
      **任务拆解**：
      1. 藏经阁：调研 Spring Security JWT 最佳实践
      2. 阵堂：实现用户认证接口
      3. 器堂：实现登录页面和状态管理
      4. 执法堂：两阶段测试验收
      
      ✅ 已创建任务：task/2026-03-29-user-login
      ✅ 已创建分支：task/2026-03-29-user-login
      ✅ 已创建任务文件：
         - 任务总览.md
         - 执行计划.md
         - 藏经阁任务.md
         - 阵堂任务.md
         - 器堂任务.md
         - 执法堂验收.md
         - .task-status.json
      
      🚀 立即开始执行...

@藏经阁长老

**任务文件**: /天玄宗/宗门任务榜/2026-03-29-user-login/藏经阁任务.md

请调研 Spring Security JWT 的最佳实践，重点关注：
- Token 生成和验证
- 刷新 Token 机制
- 安全配置最佳实践

完成后将调研结果存入功法大全。
```

### 场景二：Bug 修复

```
用户: /宗门新任务

用户: 修复用户头像上传失败的问题

宗主: 收到！让我先了解一下问题...
      
      **问题探索**：
      1. 什么情况下会失败？所有文件还是特定格式？
      2. 有错误提示吗？
      3. 最近有相关代码变更吗？

用户: 大于 2MB 的图片会失败，提示"上传超时"

宗主: 明白了！让我检查一下...
      
      @丹堂长老 请查询最近关于文件上传的日志
      
      [丹堂长老查询后]
      
      **根因分析**：
      - Nginx 上传大小限制为 1MB
      - 后端超时时间设置过短
      
      **解决方案**：
      1. 调整 Nginx client_max_body_size
      2. 增加后端上传超时时间
      3. 前端添加文件大小预检
      
      已创建任务：task/2026-03-29-fix-avatar-upload
      
      开始执行...
```

### 场景三：重构任务

```
用户: /宗门新任务

用户: 重构订单模块，代码太乱了

宗主: 收到！让我先了解一下现状...

      [查询后]

      - 订单逻辑分散在多个 Service 中

      2. 统一订单业务逻辑到 OrderService
      3. 补充单元测试
      
      **风险评估**：
      - 影响范围：订单创建、支付、发货流程
      - 建议：分阶段重构，每阶段充分测试
      
      是否继续？

用户: 继续

宗主: 好的！已创建任务：task/2026-03-29-refactor-order
      
      开始执行...
```

## 配置选项

```bash
# 快速模式（跳过详细探索）
/宗门新任务 --quick

# 指定任务类型
/宗门新任务 --type feature    # 新功能
/宗门新任务 --type bugfix     # Bug 修复
/宗门新任务 --type refactor   # 重构

# 指定优先级
/宗门新任务 --priority high   # 高优先级
```

## 注意事项

1. **需求必须明确**
   - 宗主会充分探索需求
   - 不明确的需求会要求用户补充

2. **遵循 HARD-GATE 原则**
   - Never guess（不猜测）
   - Never use placeholders（不用占位符）
   - Never skip steps（不跳过步骤）

3. **任务隔离**
   - 每个任务独立的 Git 分支
   - 任务完成后才能合并

4. **质量保证**
   - 执法堂两阶段测试
   - 代码审查必须通过
