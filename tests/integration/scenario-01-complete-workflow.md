# 集成测试场景 01: 完整工作流

## 场景描述

测试从需求探索到归档的完整工作流程。

## 测试目标

验证天玄宗多 Agent 协作系统的端到端流程。

## 前置条件

1. 天玄宗插件已安装并启用
2. 已执行 `/init-tianxuan` 初始化项目
3. 有一个干净的测试项目环境

## 测试步骤

### Phase 1: 初始化 (5分钟)

```bash
# 1. 初始化天玄宗
/init-tianxuan

# 2. 验证目录结构
ls -la 天玄宗/
# 应该看到: 宗门任务榜/, 藏经阁/, 丹堂/, 阵堂/, 器堂/, 执法堂/

# 3. 验证配置文件
cat 天玄宗/.tianxuan.config.json
cat 天玄宗/藏经阁/INDEX.md
```

**预期结果**:
- ✅ 所有目录创建成功
- ✅ 配置文件存在且格式正确
- ✅ 索引文件存在

---

### Phase 2: 需求探索 (15分钟)

```
用户: @宗主 我想实现一个简单的待办事项管理功能

宗主: (应该开始提问)

用户: (回答宗主的问题)
- 技术栈: Node.js + Express (后端), React (前端)
- 数据库: SQLite
- 功能: 创建、查看、更新、删除待办事项
```

**预期行为**:
1. ✅ 宗主逐步提问了解需求
2. ✅ 宗主委派藏经阁调研 Express、React 最佳实践
3. ✅ 宗主提出设计方案（数据库结构、API 设计）
4. ✅ 宗主分段展示设计并等待确认
5. ❌ 在确认前**不创建**任务文件夹

---

### Phase 3: 设计确认 (5分钟)

```
用户: 确认设计方案
```

**预期行为**:
1. ✅ 宗主创建任务分支 `task/YYYY-MM-DD-todo-management`
2. ✅ 宗主创建任务文件夹 `天玄宗/宗门任务榜/YYYY-MM-DD-todo-management/`
3. ✅ 生成以下文件:
   - 任务总览.md
   - .task-status.json
   - 藏经阁详细任务.md
   - 阵堂详细任务.md
   - 器堂详细任务.md

---

### Phase 4: 藏经阁调研 (10分钟)

**预期行为**:
1. ✅ 宗主调用 `@藏经阁长老`
2. ✅ 藏经阁查询 Express、React 相关文档
3. ✅ 藏经阁总结并存入 `天玄宗/藏经阁/技术总结/`
4. ✅ 藏经阁更新 `INDEX.md`
5. ✅ 藏经阁通知宗主完成

**验证**:
```bash
# 检查藏经阁输出
ls 天玄宗/藏经阁/技术总结/
cat 天玄宗/藏经阁/INDEX.md
```

---

### Phase 5: 阵堂开发 (20分钟)

**预期行为**:
1. ✅ 宗主调用 `@阵堂长老`
2. ✅ 阵堂先查询藏经阁 (藏经阁优先原则)
3. ✅ 阵堂读取任务文件
4. ✅ 阵堂实现后端代码
5. ✅ 阵堂执行 Self-Review 自检
6. ✅ 阵堂通知宗主完成

**验证**:
```bash
# 检查后端代码
ls src/
cat src/routes/todos.js
cat src/models/todo.js

# 检查参数校验
grep -r "StringUtils\|Objects.requireNonNull" src/

# 检查异常处理
grep -r "try.*catch" src/
```

---

### Phase 6: 器堂开发 (20分钟)

**预期行为**:
1. ✅ 宗主调用 `@器堂长老`
2. ✅ 器堂先查询藏经阁
3. ✅ 器堂读取任务文件
4. ✅ 器堂实现前端代码
5. ✅ 器堂执行 Self-Review 自检
6. ✅ 器堂通知宗主完成

**验证**:
```bash
# 检查前端代码
ls src/components/
cat src/components/TodoList.jsx
cat src/api/todos.js

# 检查错误处理
grep -r "try.*catch\|\.catch" src/components/
```

---

### Phase 7: 执法堂测试 (30分钟)

**预期行为**:

**阶段1: 功能符合性测试**
1. ✅ 执法堂读取验收标准
2. ✅ 执法堂逐项验证功能
3. ✅ 执法堂生成 `执法堂-功能符合性报告.md`

**阶段2: 代码质量测试**
1. ✅ 执法堂进行边界测试
2. ✅ 执法堂进行异常测试
3. ✅ 执法堂进行性能测试
4. ✅ 执法堂进行安全测试
5. ✅ 执法堂生成 `执法堂-质量测试报告.md`

**如果测试失败**:
1. ✅ 执法堂生成诊断报告
2. ✅ 执法堂打回对应的堂修复
3. ✅ 失败计数 +1
4. ✅ 如果失败 3 次，通知宗主质疑架构

**验证**:
```bash
# 检查测试报告
ls 天玄宗/宗门任务榜/*/执法堂-*.md
cat 天玄宗/宗门任务榜/*/执法堂-功能符合性报告.md
cat 天玄宗/宗门任务榜/*/执法堂-质量测试报告.md
```

---

### Phase 8: 归档 (5分钟)

**预期行为**:
1. ✅ 宗主合并分支到 main
2. ✅ 宗主删除任务分支
3. ✅ 宗主移动任务到归档
4. ✅ 宗主生成经验总结
5. ✅ 宗主更新藏经阁索引

**验证**:
```bash
# 检查分支
git branch
# 应该看不到 task/YYYY-MM-DD-todo-management

# 检查归档
ls 天玄宗/藏经阁/归档/
cat 天玄宗/藏经阁/项目经验/*-总结.md

# 检查索引
cat 天玄宗/藏经阁/INDEX.md
```
