# Brainstorming (需求探索)

## 技能说明

在任何创造性工作之前，通过提问、探索替代方案、分段展示设计并获得用户批准，将想法转化为完整的设计和规范。

## 核心原则

**先理解再行动** - 在设计获得用户批准之前，不得进行任何实现、编写代码或创建任务。

## 执行流程

### 1. 探索项目上下文

在开始设计之前，先了解项目背景：

- **检查现有代码**: 了解当前代码结构、技术栈
- **查看配置文件**: 了解项目配置、依赖
- **查看 git 历史**: 了解最近的变更
- **查看文档**: 了解灵脉信息、API 文档

### 2. 委派藏经阁调研

如果涉及第三方平台或新技术，先让藏经阁调研：

```
@藏经阁长老

**任务**: 调研 [相关技术/平台] 的官方文档

**重点关注**:
- [关键点1]
- [关键点2]
- [关键点3]

请总结关键 API、参数、示例代码，并存入藏经阁。
```

### 3. 逐步厘清需求

**一次问一个问题**，不要一次性问太多：

```
问题1: 您使用的技术栈是什么？
(等待用户回答)

问题2: 数据库用什么？
(等待用户回答)

问题3: 有什么特殊要求吗？
(等待用户回答)
```

### 4. 提出多种方案

基于调研和需求，提出 2-3 种方案：

```markdown
基于藏经阁的调研，我给您几个方案：

**方案A**: [描述]
- 优点: [优点]
- 缺点: [缺点]
- 适用场景: [场景]

**方案B**: [描述]
- 优点: [优点]
- 缺点: [缺点]
- 适用场景: [场景]

我推荐**方案A**，因为 [理由]。

您倾向于哪个方案？或者有其他想法？
```

### 5. 分段展示设计

**不要一次性展示完整设计**，而是分段展示，每个部分确认后再继续：

```markdown
## 数据库设计

### 表1: user
- id: BIGINT PRIMARY KEY
- username: VARCHAR(50) NOT NULL
- email: VARCHAR(100) NOT NULL
- created_at: DATETIME NOT NULL

### 表2: user_profile
- id: BIGINT PRIMARY KEY
- user_id: BIGINT NOT NULL
- avatar: VARCHAR(200)
- bio: TEXT

这样可以吗？

(等待用户确认)
```

确认后继续下一部分：

```markdown
## 接口设计

### POST /api/users
创建用户

**请求参数**:
- username: string (必填)
- email: string (必填)
- password: string (必填)

**返回**:
- code: number
- message: string
- data: { userId: string }

### GET /api/users/:id
获取用户信息

**请求参数**:
- id: string (路径参数)

**返回**:
- code: number
- message: string
- data: { user: {...} }

这样可以吗？

(等待用户确认)
```

### 6. 获得用户确认

**必须获得明确的确认**，才能进入下一阶段：

- ✅ "确认"
- ✅ "同意"
- ✅ "可以"
- ✅ "没问题"
- ✅ "就这样"

**不算确认**：
- ❌ "嗯"
- ❌ "好的"（可能只是礼貌性回应）
- ❌ 沉默

如果用户提出修改意见，重新调整设计并再次确认。

## HARD-GATE 检查点

**在用户明确确认设计之前，禁止**：

- ❌ 创建任务文件夹
- ❌ 生成任务文件
- ❌ 编写代码
- ❌ 修改现有代码
- ❌ 创建分支

**只有在用户明确确认后，才能**：

- ✅ 进入任务拆解阶段
- ✅ 创建任务文件夹
- ✅ 生成任务文件

## 示例对话

```
宗主: 您想实现什么功能？

用户: 我想做一个用户管理系统

宗主: 好的，我先了解一下项目背景。您使用的技术栈是什么？

用户: 后端用 Spring Boot，前端用 Vue 3

宗主: 明白了。数据库用什么？

用户: MySQL 8.0

宗主: 好的。@藏经阁长老 查询 Spring Boot 用户管理最佳实践

藏经阁长老: 已完成，文档存入藏经阁

宗主: 基于藏经阁的调研，我给您两个方案：

**方案A**: 使用 Spring Security + JWT
- 优点: 成熟稳定，社区支持好
- 缺点: 配置稍复杂

**方案B**: 使用 Session + Cookie
- 优点: 简单易用
- 缺点: 不适合分布式

我推荐**方案A**，因为更适合现代应用。您倾向于哪个方案？

用户: 用方案A

宗主: 好的。我先设计数据库结构：

## 数据库设计

### user 表
- id: BIGINT PRIMARY KEY AUTO_INCREMENT
- username: VARCHAR(50) NOT NULL UNIQUE
- password: VARCHAR(200) NOT NULL
- email: VARCHAR(100) NOT NULL UNIQUE
- status: TINYINT NOT NULL DEFAULT 1
- created_at: DATETIME NOT NULL
- updated_at: DATETIME NOT NULL

这样可以吗？

用户: 可以

宗主: 好的。接下来是接口设计：

## 接口设计

### POST /api/auth/register
用户注册
...

### POST /api/auth/login
用户登录
...

这样可以吗？

用户: 确认

宗主: 好的，设计已确认。我现在开始创建任务。
```

## 注意事项

1. **不要假设用户需求** - 通过提问确认
2. **不要一次性展示所有设计** - 分段展示
3. **不要跳过用户确认** - 必须明确确认
4. **不要在未确认前实现** - HARD-GATE 原则
