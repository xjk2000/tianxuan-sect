# 灵脉勘查命令

## 概述

`/灵脉勘查` 命令深入分析项目结构、技术栈、架构设计，并将勘查所得知识按**项目名称**存储到独立子目录 `天玄宗/藏经阁/灵脉信息/{项目名}/`，支持多项目独立管理，互不干扰。

---

## 执行流程

### 第零阶段：识别工作区类型 + 确认勘查目标

**第一步：检测工作区类型（单项目 vs 多子项目）**

宗主必须先判断当前工作区是单项目还是多子项目工作区（mono-repo）：

```bash
# === 辅助函数1：检测目录是否是一个项目（支持所有主流语言/构建工具）===
is_project_dir() {
  local dir="$1"
  # 强标志：各语言官方项目配置文件
  [ -f "$dir/package.json" ]    && return 0  # Node.js / 前端（JS/TS/Vue/React）
  [ -f "$dir/pom.xml" ]         && return 0  # Java Maven
  [ -f "$dir/build.gradle" ]    && return 0  # Java/Android Gradle
  [ -f "$dir/build.gradle.kts" ] && return 0 # Kotlin Gradle
  [ -f "$dir/pyproject.toml" ]  && return 0  # Python（Poetry/PDM/Hatch/PEP 517+）
  [ -f "$dir/setup.py" ]        && return 0  # Python（setuptools 旧式）
  [ -f "$dir/setup.cfg" ]       && return 0  # Python（setuptools 旧式）
  [ -f "$dir/go.mod" ]          && return 0  # Go
  [ -f "$dir/Cargo.toml" ]      && return 0  # Rust
  [ -f "$dir/CMakeLists.txt" ]  && return 0  # C++ / C（CMake）
  [ -f "$dir/Makefile" ]        && return 0  # C/C++/通用 make 项目
  [ -f "$dir/composer.json" ]   && return 0  # PHP（Composer）
  [ -f "$dir/Gemfile" ]         && return 0  # Ruby（Bundler）
  [ -f "$dir/mix.exs" ]         && return 0  # Elixir（Mix）
  [ -f "$dir/build.sbt" ]       && return 0  # Scala（SBT）
  [ -f "$dir/pubspec.yaml" ]    && return 0  # Dart / Flutter
  [ -f "$dir/Package.swift" ]   && return 0  # Swift（SPM）
  [ -f "$dir/project.clj" ]     && return 0  # Clojure（Leiningen）
  # .NET/C#：glob 匹配 *.csproj / *.sln / *.fsproj
  ls "$dir"/*.csproj 2>/dev/null | grep -q . && return 0
  ls "$dir"/*.sln    2>/dev/null | grep -q . && return 0
  ls "$dir"/*.fsproj 2>/dev/null | grep -q . && return 0
  return 1
}

# === 辅助函数2：提取项目名（无法解析时统一兜底目录名）===
extract_project_name() {
  local dir="${1:-.}"
  local name=""

  if [ -f "$dir/package.json" ]; then
    name=$(grep '"name"' "$dir/package.json" | head -1 \
      | sed 's/.*"name" *: *"\([^"]*\)".*/\1/' | tr -d '[:space:]')
  elif [ -f "$dir/pom.xml" ]; then
    name=$(grep -m1 '<artifactId>' "$dir/pom.xml" \
      | sed 's/.*<artifactId>\([^<]*\)<\/artifactId>.*/\1/' | tr -d '[:space:]')
  elif [ -f "$dir/pyproject.toml" ]; then
    name=$(grep '^name' "$dir/pyproject.toml" | head -1 \
      | sed 's/name *= *"\([^"]*\)"/\1/' | tr -d '[:space:]')
  elif [ -f "$dir/setup.py" ]; then
    name=$(grep -m1 'name=' "$dir/setup.py" \
      | sed "s/.*name=['\"\(]*\([^'\",)]*\).*/\1/" | tr -d '[:space:]')
  elif [ -f "$dir/go.mod" ]; then
    name=$(grep '^module' "$dir/go.mod" | head -1 | awk '{print $2}' | xargs basename)
  elif [ -f "$dir/Cargo.toml" ]; then
    name=$(grep '^name' "$dir/Cargo.toml" | head -1 \
      | sed 's/name *= *"\([^"]*\)"/\1/' | tr -d '[:space:]')
  elif [ -f "$dir/composer.json" ]; then
    # composer name 格式为 "vendor/package"，取 package 部分
    name=$(grep '"name"' "$dir/composer.json" | head -1 \
      | sed 's/.*"name" *: *"[^\/]*\/\([^"]*\)".*/\1/' | tr -d '[:space:]')
  elif [ -f "$dir/pubspec.yaml" ]; then
    name=$(grep '^name:' "$dir/pubspec.yaml" | head -1 \
      | sed 's/name: *//' | tr -d '[:space:]')
  fi

  # ⚠️ 兜底：C++/Make/Ruby/Elixir/Scala/.NET 等无法解析名称时，统一用目录名
  [ -z "$name" ] && name=$(basename "$dir")

  echo "$name"
}

# === 第一阶段：判断工作区类型 ===

# 排除列表：不视为子项目的目录
EXCLUDE_DIRS="天玄宗 node_modules .git dist build target .idea .vscode __pycache__ .cache"

# 检查根目录是否直接是一个项目
if is_project_dir "."; then
  # ✅ 根目录就是单项目
  WORKSPACE_TYPE="single"
  ROOT_PROJECT_NAME=$(extract_project_name ".")
  echo "工作区类型：单项目（$ROOT_PROJECT_NAME）"
else
  # 扫描子目录，查找子项目
  SUBPROJECTS=()
  for d in */; do
    dir="${d%/}"
    skip=0
    for ex in $EXCLUDE_DIRS; do [ "$dir" = "$ex" ] && skip=1; done
    [ $skip -eq 1 ] && continue
    is_project_dir "$dir" && SUBPROJECTS+=("$dir")
  done

  if [ ${#SUBPROJECTS[@]} -eq 0 ]; then
    # 没有找到任何子项目，兜底用目录名
    WORKSPACE_TYPE="single"
    ROOT_PROJECT_NAME=$(basename "$(pwd)")
    echo "工作区类型：单项目（兜底目录名 $ROOT_PROJECT_NAME）"
  elif [ ${#SUBPROJECTS[@]} -eq 1 ]; then
    WORKSPACE_TYPE="single"
    ROOT_PROJECT_NAME=$(extract_project_name "${SUBPROJECTS[0]}")
    echo "工作区类型：单子项目（$ROOT_PROJECT_NAME）"
  else
    # 多个子项目 → mono-repo
    WORKSPACE_TYPE="monorepo"
    echo "工作区类型：多子项目工作区"
    echo "检测到以下子项目："
    for i in "${!SUBPROJECTS[@]}"; do
      pname=$(extract_project_name "${SUBPROJECTS[$i]}")
      echo "  $((i+1)). ${SUBPROJECTS[$i]}  →  文档目录: 天玄宗/藏经阁/灵脉信息/$pname/"
    done
  fi
fi
```

### 第一阶段：宗主主导项目分析

1. **扫描项目结构**
   - 识别当前目录结构，找到根目录
   - 识别关键配置文件（`package.json`, `pom.xml`, `requirements.txt`, `go.mod` 等）
   - 检测技术栈和框架版本

2. **分析技术架构**
   - 识别前端/后端/全栈技术栈
   - 分析项目分层结构（Controller/Service/Repository 等）
   - 识别核心模块和依赖关系
   - 检测设计模式和架构风格

3. **提取关键信息**
   - 项目功能定位和描述
   - 主要功能模块清单
   - API 端点和路由规律
   - 数据模型和数据库表结构
   - 关键配置和环境变量

4. **识别开发规范**
   - 代码风格和格式化规则
   - 命名约定（包名、类名、接口名等）
   - 目录组织规范
   - 测试策略和覆盖率

5. **收集环境配置信息**

   宗主**必须**与无上道友确认以下环境配置，存入该项目的流程规范：

   **后端环境（阵堂）**：
   - 框架和版本（如 Spring Boot 3.2）
   - 语言版本（如 Java 17）
   - 构建工具和命令（如 `mvn spring-boot:run`）
   - 服务端口（如 8080）

   **前端环境（器堂）**：
   - 框架和版本（如 Vue 3 / React 18）
   - 包管理器（npm / yarn / pnpm）
   - 启动命令（如 `pnpm dev`）
   - 开发服务器端口（如 5173）

   **数据库配置（丹堂）**：
   - 数据库类型和版本（如 MySQL 8.0）
   - 连接地址（host:port/db）
   - 是否有 Redis/MongoDB 等其他存储

   **测试环境（执法堂）**：
   - 测试框架和命令（如 `mvn test`）
   - 测试覆盖率要求

```bash
  "password": "[实际password]",
  "charset": "utf8mb4"

EOF

# 添加到 .gitignore
echo "天玄宗/丹堂/工具/$PROJECT_NAME/db_config.json" >> .gitignore
```

### 场景二：多子项目工作区（mono-repo）

```
用户: /灵脉勘查

宗主: 🔍 检测工作区类型...
      根目录无项目配置文件，扫描子目录...
      
      检测到多子项目工作区（mono-repo），包含：
      1. shulex-intelli     → 天玄宗/藏经阁/灵脉信息/shulex-intelli/
         (Java Maven 多模块后端)
      2. shulex-smart-service → 天玄宗/藏经阁/灵脉信息/shulex-smart-service/
```

## 注意事项

1. **项目名唯一性**：同一工作区中项目名不能重复，如有冲突宗主会询问确认
2. **增量更新保留历史**：增量更新只修改变更文档，不删除已有内容
3. **流程规范按项目隔离**：不同项目的启动命令、端口配置互不干扰
4. **敏感信息保护**：`db_config.json` 自动加入 `.gitignore`，不提交到版本库
5. **与 /宗门初始化 的区别**：`/宗门初始化` 创建天玄宗目录结构；`/灵脉勘查` 勘查具体项目并生成文档

---

**提示**: 每个项目都有独立的灵脉文档目录，多项目并行开发时各自文档清晰隔离，不会互相污染！
