# 天玄宗快速入门指南

欢迎使用天玄宗修仙宗门多 Agent 协作架构！

## 什么是天玄宗？

天玄宗是一个专为软件开发设计的多 Agent 协作系统，通过 6 个专业角色的分工协作，实现高效、高质量的软件开发。

### 核心特性

- 🎯 **专业分工** - 6 个专门的 Agent 各司其职
- 📚 **知识管理** - 藏经阁机制实现经验复用
- 🔒 **质量保证** - HARD-GATE + 两阶段审查 + Self-Review
- 🚫 **No Placeholders** - 任务定义必须完整可执行
- 🔄 **隔离上下文** - 降低 Token 消耗
- ⚙️ **配置驱动** - 灵活适配任何技术栈

## 安装

```bash

# 查看可用的 agents
```

## 初始化项目

在你的项目根目录执行：

```bash
/init-tianxuan
```

这将创建以下目录结构：

```
天玄宗/
├── 宗门任务榜/              # 活跃任务
├── 藏经阁/                  # 知识库
│   ├── INDEX.md            # 知识索引
│   ├── 归档/               # 已完成任务
│   ├── 第三方平台/         # 第三方 API 文档
│   ├── 项目经验/           # 项目经验总结
│   └── 技术总结/           # 技术栈最佳实践
├── 丹堂/查询脚本/
├── 阵堂/临时文件/
├── 器堂/临时文件/
└── 执法堂/测试脚本/
```

## 第一个任务

### 1. 向宗主下达任务

```
@宗主 我想实现一个用户注册功能
```

### 2. 需求探索

宗主会开始提问了解需求：

```
宗主: 好的，我先了解一下项目背景。您使用的技术栈是什么？
```

回答宗主的问题：

```
用户: 后端用 Node.js + Express，前端用 React
```

### 3. 技术调研

宗主会委派藏经阁调研相关技术：

```
宗主: @藏经阁长老 查询 Express 用户认证最佳实践
```

### 4. 设计方案

宗主会提出设计方案：

```
宗主: 基于藏经阁的调研，我给您两个方案：

**方案A**: 使用 JWT + bcrypt
- 优点: 无状态，易于扩展
- 缺点: 需要客户端存储 token

**方案B**: 使用 Session + Cookie
- 优点: 服务端控制，更安全
- 缺点: 需要 session 存储

我推荐**方案A**，因为更适合前后端分离架构。您倾向于哪个方案？
```

### 5. 确认设计

```
用户: 用方案A
```

### 6. 分段展示设计

宗主会分段展示详细设计：

```
宗主: 好的。我先设计数据库结构：

## 数据库设计

### user 表
- id: VARCHAR(36) PRIMARY KEY
- username: VARCHAR(50) NOT NULL UNIQUE
- email: VARCHAR(100) NOT NULL UNIQUE
- password_hash: VARCHAR(200) NOT NULL
- created_at: DATETIME NOT NULL

这样可以吗？
```

确认每个部分：

```
用户: 可以
```

### 7. 任务创建

设计确认后，宗主会创建任务：

```
宗主: 设计已确认，开始创建任务

(创建任务分支和文件)

任务已创建: 2026-03-28-user-registration
```

### 8. 各堂协作

宗主会依次调度各堂：

1. **藏经阁** - 调研并存储相关文档
2. **阵堂** - 实现后端接口
3. **器堂** - 实现前端组件
4. **执法堂** - 两阶段测试

### 9. 查看任务状态

```bash
/task-status
```

输出：

```
📋 当前活跃任务:

## 2026-03-28-user-registration

**状态**: in_progress
**创建时间**: 2026-03-28T12:00:00Z
**分支**: task/2026-03-28-user-registration

- 藏经阁: ✅ 已完成

- 执法堂: ⏸️ 等待中 (阻塞: 阵堂, 器堂)
```

执法堂测试通过后，宗主会自动归档：

## 核心概念

### HARD-GATE

宗主在用户确认设计前，**禁止**创建任务文件夹。

这确保了：
- ✅ 需求充分探索
- ✅ 设计经过深思熟虑
- ✅ 用户完全理解方案

### No Placeholders

任务文件必须包含完整定义，**禁止**模糊描述：

❌ 错误示例：
```
- 添加适当的参数校验
- 处理异常情况
- 实现相关功能
```

✅ 正确示例：
```
- 参数校验：
  - username: 长度 3-50，只允许字母数字下划线
  - email: 必须是有效的邮箱格式
  - password: 长度至少 8 字符
- 异常处理：
  - 用户名重复 → 返回 400，提示"用户名已存在"
  - 邮箱重复 → 返回 400，提示"邮箱已被注册"
  - 数据库错误 → 返回 500，提示"系统异常"
```

### 藏经阁优先

各堂在开始任务前，**必须**先查询藏经阁：

```
@藏经阁长老 查询以下文档:
- Express 最佳实践
- JWT 认证实现
- 密码加密方案
```

这确保了：
- ✅ 复用已有知识
- ✅ 遵循统一规范
- ✅ 避免重复调研

### Self-Review

各堂完成后执行自检清单：

```
Self-Review 自检:

✅ 参数校验完整
✅ 异常处理完整
✅ 事务管理正确
✅ 日志记录完整
✅ 魔法值已提取
✅ 代码符合规范
✅ 没有 TODO/FIXME
✅ 没有调试代码
```

### 两阶段审查

执法堂进行两阶段测试：

**阶段1: 功能符合性测试**
- 验证是否实现了验收标准
- 逐项检查功能

**阶段2: 代码质量测试**
- 边界测试
- 异常测试
- 性能测试
- 安全测试

### Systematic Debugging

测试失败时：

1. **根因诊断** - 找到失败的根本原因
2. **打回修复** - 通知责任堂修复
3. **3次失败** - 通知宗主质疑架构

## 常见问题

### Q: 如何修改 Agent 使用的模型？

A: 编辑 `天玄宗/.tianxuan.config.json`：

```json
{
  "agents": {
    "宗主": {
      "model": {
        "primary": "opus",
        "fallbacks": ["sonnet"]
      }
    }
  }
}
```

### Q: 如何查看当前任务状态？

A: 使用 `/task-status` 命令。

### Q: 执法堂测试失败怎么办？

A: 查看诊断报告，根据修复建议修复问题，然后通知执法堂重新测试。

### Q: 如何手动归档任务？

A: 使用 `/archive-task <taskId>` 命令。

### Q: 可以跳过某个堂吗？

A: 不建议。每个堂都有其职责，跳过可能导致质量问题。
