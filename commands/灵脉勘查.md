# 灵脉勘查命令

## 概述

`/灵脉勘查` 命令深入分析项目结构、技术栈、架构设计，并将勘查所得知识按**项目名称**存储到独立子目录 `天玄宗/藏经阁/灵脉信息/{项目名}/`，支持多项目独立管理，互不干扰。

---

## 执行流程

### 第零阶段：识别项目名 + 检查历史勘查

**第一步：自动识别项目名称**

宗主必须先识别项目名，作为后续所有文档的命名空间：

```bash
# 按优先级自动提取项目名
PROJECT_NAME=""

# 优先级1: package.json → name 字段
if [ -f "package.json" ]; then
  PROJECT_NAME=$(grep '"name"' package.json | head -1 \
    | sed 's/.*"name" *: *"\([^"]*\)".*/\1/' | tr -d '[:space:]')
fi

# 优先级2: pom.xml → 第一个 artifactId（跳过 parent）
if [ -z "$PROJECT_NAME" ] && [ -f "pom.xml" ]; then
  PROJECT_NAME=$(grep -m1 '<artifactId>' pom.xml \
    | sed 's/.*<artifactId>\([^<]*\)<\/artifactId>.*/\1/' | tr -d '[:space:]')
fi

# 优先级3: pyproject.toml → name
if [ -z "$PROJECT_NAME" ] && [ -f "pyproject.toml" ]; then
  PROJECT_NAME=$(grep '^name' pyproject.toml | head -1 \
    | sed 's/name *= *"\([^"]*\)"/\1/' | tr -d '[:space:]')
fi

# 优先级4: go.mod → module 最后一段
if [ -z "$PROJECT_NAME" ] && [ -f "go.mod" ]; then
  PROJECT_NAME=$(grep '^module' go.mod | head -1 | awk '{print $2}' | xargs basename)
fi

# 优先级5: 兜底使用当前目录名
if [ -z "$PROJECT_NAME" ]; then
  PROJECT_NAME=$(basename "$(pwd)")
fi

echo "识别到项目名: $PROJECT_NAME"
```

**向无上道友确认项目名**（如自动识别不准确可修正）：

```
宗主: 已识别项目名为「$PROJECT_NAME」，文档将存储到：
      天玄宗/藏经阁/灵脉信息/$PROJECT_NAME/
      
      确认此名称，还是需要修正？
```

**第二步：检查历史勘查记录**

```bash
PROJECT_DOC_DIR="天玄宗/藏经阁/灵脉信息/$PROJECT_NAME"
PROJECT_RULE_DIR="天玄宗/宗门流程规范/$PROJECT_NAME"

# 检查是否有历史勘查
if [ -d "$PROJECT_DOC_DIR" ]; then
  echo "⚠️ 已有「$PROJECT_NAME」的灵脉信息："
  ls "$PROJECT_DOC_DIR/"
  echo "上次更新：$(ls -lt $PROJECT_DOC_DIR | head -2 | tail -1 | awk '{print $6,$7,$8}')"
fi

# 天玄记忆诀：检索历史观照
grep -i "$PROJECT_NAME\|灵脉\|架构\|技术栈" 天玄宗/道行观照/OBSERVATIONS.md 2>/dev/null | head -10
```

- 如果**已有历史勘查**：告知用户上次结果，询问是否**增量更新**还是**全量重勘**
- 如果**首次勘查**：直接进入完整勘查流程

### 第一阶段：宗主主导项目分析

1. **扫描项目结构**
   - 识别当前目录结构，找到根目录
   - 识别关键配置文件（`package.json`, `pom.xml`, `requirements.txt`, `go.mod` 等）
   - 检测技术栈和框架版本

2. **分析技术架构**
   - 识别前端/后端/全栈技术栈
   - 分析项目分层结构（Controller/Service/Repository 等）
   - 识别核心模块和依赖关系
   - 检测设计模式和架构风格

3. **提取关键信息**
   - 项目功能定位和描述
   - 主要功能模块清单
   - API 端点和路由规律
   - 数据模型和数据库表结构
   - 关键配置和环境变量

4. **识别开发规范**
   - 代码风格和格式化规则
   - 命名约定（包名、类名、接口名等）
   - 目录组织规范
   - 测试策略和覆盖率

5. **收集环境配置信息**

   宗主**必须**与无上道友确认以下环境配置，存入该项目的流程规范：

   **后端环境（阵堂）**：
   - 框架和版本（如 Spring Boot 3.2）
   - 语言版本（如 Java 17）
   - 构建工具和命令（如 `mvn spring-boot:run`）
   - 服务端口（如 8080）

   **前端环境（器堂）**：
   - 框架和版本（如 Vue 3 / React 18）
   - 包管理器（npm / yarn / pnpm）
   - 启动命令（如 `pnpm dev`）
   - 开发服务器端口（如 5173）

   **数据库配置（丹堂）**：
   - 数据库类型和版本（如 MySQL 8.0）
   - 连接地址（host:port/db）
   - 是否有 Redis/MongoDB 等其他存储

   **测试环境（执法堂）**：
   - 测试框架和命令（如 `mvn test`）
   - 测试覆盖率要求

### 第二阶段：按项目创建宗门流程规范

宗主根据收集的配置信息，在**项目专属目录**创建流程规范：

```bash
# 创建项目专属流程规范目录
mkdir -p "天玄宗/宗门流程规范/$PROJECT_NAME"

# 从模板复制并命名
cp tianxuan-sect/templates/阵堂规范.md "天玄宗/宗门流程规范/$PROJECT_NAME/阵堂规范.md"
cp tianxuan-sect/templates/器堂规范.md "天玄宗/宗门流程规范/$PROJECT_NAME/器堂规范.md"
cp tianxuan-sect/templates/丹堂规范.md "天玄宗/宗门流程规范/$PROJECT_NAME/丹堂规范.md"
cp tianxuan-sect/templates/执法堂规范.md "天玄宗/宗门流程规范/$PROJECT_NAME/执法堂规范.md"
```

宗主使用 Edit 工具将确认的配置信息填入各规范文档，替换所有 `[待填写]` 占位符。

```bash
# 创建项目专属数据库配置（保护敏感信息）
mkdir -p "天玄宗/丹堂/工具/$PROJECT_NAME"
cat > "天玄宗/丹堂/工具/$PROJECT_NAME/db_config.json" << EOF
{
  "project": "$PROJECT_NAME",
  "type": "[mysql/postgresql/mongodb]",
  "host": "[实际host]",
  "port": [实际port],
  "database": "[实际database]",
  "user": "[实际user]",
  "password": "[实际password]",
  "charset": "utf8mb4"
}
EOF

# 添加到 .gitignore
echo "天玄宗/丹堂/工具/$PROJECT_NAME/db_config.json" >> .gitignore
```

## 配置选项

```bash
# 指定勘查目录（默认当前目录）
/灵脉勘查 ../my-backend

# 手动指定项目名（覆盖自动识别）
/灵脉勘查 --project my-custom-name

# 仅更新特定文档
/灵脉勘查 --update api
/灵脉勘查 --update database

# 快速勘查（仅基本信息，跳过环境配置收集）
/灵脉勘查 --quick
```

## 注意事项

1. **项目名唯一性**：同一工作区中项目名不能重复，如有冲突宗主会询问确认
2. **增量更新保留历史**：增量更新只修改变更文档，不删除已有内容
3. **流程规范按项目隔离**：不同项目的启动命令、端口配置互不干扰
4. **敏感信息保护**：`db_config.json` 自动加入 `.gitignore`，不提交到版本库
5. **与 /宗门初始化 的区别**：`/宗门初始化` 创建天玄宗目录结构；`/灵脉勘查` 勘查具体项目并生成文档

---

**提示**: 每个项目都有独立的灵脉文档目录，多项目并行开发时各自文档清晰隔离，不会互相污染！
