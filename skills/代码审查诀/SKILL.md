# Code Review (代码审查)

## 技能说明

为天玄宗提供系统化的代码审查流程，确保代码质量和团队协作。

## 自动化审查工具

使用自动化工具辅助代码审查：

```bash
# JavaScript/TypeScript 文件审查
node review-checklist.js src/user.js

# 检查项目：
# - 命名规范
# - 函数长度
# - 注释比例
# - 错误处理
# - 安全问题
# - 性能问题
```

详见 [review-checklist.js](review-checklist.js)。

## 核心原则

**建设性反馈** - 代码审查的目的是提高代码质量，而不是批评开发者。

## 审查时机

### 执法堂审查

执法堂在两阶段测试中会进行代码审查：
- **阶段1**: 功能符合性（是否实现了需求）
- **阶段2**: 代码质量（是否符合标准）

### 宗主审查

宗主在归档前会进行最终审查：
- 检查任务完整性
- 验证文档完整性
- 确认知识沉淀

## 审查清单

### 1. 代码风格

#### 命名规范

- [ ] **变量命名** - 语义清晰，使用驼峰命名
- [ ] **函数命名** - 动词开头，描述功能
- [ ] **类命名** - 名词，使用 PascalCase
- [ ] **常量命名** - 全大写，下划线分隔

#### 格式规范

- [ ] **缩进一致** - 使用统一的缩进（2空格或4空格）
- [ ] **行长度** - 不超过 100 字符
- [ ] **空行使用** - 逻辑块之间有空行
- [ ] **括号风格** - 统一的括号风格

### 2. 代码质量

#### 可读性

- [ ] **代码简洁** - 避免复杂的嵌套和长函数
- [ ] **注释充分** - 复杂逻辑有注释说明
- [ ] **魔法值消除** - 所有魔法值已提取为常量
- [ ] **重复代码** - 没有明显的重复代码

#### 健壮性

- [ ] **参数校验** - 所有入参都有校验
- [ ] **异常处理** - 所有可能的异常都有处理
- [ ] **边界情况** - 考虑了边界情况（空值、0、负数等）
- [ ] **资源释放** - 资源（文件、连接）正确释放

#### 性能

- [ ] **算法效率** - 使用合适的算法和数据结构
- [ ] **数据库查询** - 避免 N+1 查询
- [ ] **缓存使用** - 合理使用缓存
- [ ] **资源占用** - 避免内存泄漏

### 3. 安全性

#### 输入验证

- [ ] **SQL 注入** - 使用参数化查询
- [ ] **XSS 防护** - 输出转义
- [ ] **CSRF 防护** - 使用 CSRF token
- [ ] **文件上传** - 验证文件类型和大小

#### 敏感信息

- [ ] **密码加密** - 密码不明文存储
- [ ] **敏感信息** - 不在日志中记录敏感信息
- [ ] **API 密钥** - 不硬编码 API 密钥
- [ ] **错误信息** - 不暴露系统信息

### 4. 测试

#### 测试覆盖

- [ ] **单元测试** - 核心逻辑有单元测试
- [ ] **集成测试** - 关键流程有集成测试
- [ ] **边界测试** - 边界情况有测试
- [ ] **异常测试** - 异常场景有测试

#### 测试质量

- [ ] **测试独立** - 测试之间相互独立
- [ ] **测试清晰** - 测试意图清晰
- [ ] **测试完整** - 覆盖主要场景
- [ ] **测试可维护** - 测试代码易于维护

### 5. 文档

#### 代码文档

- [ ] **函数注释** - 公共函数有 Javadoc/JSDoc
- [ ] **类注释** - 类有说明注释
- [ ] **复杂逻辑** - 复杂逻辑有解释
- [ ] **TODO 清理** - 没有未处理的 TODO

#### 外部文档

- [ ] **README 更新** - 如有新功能，更新 README
- [ ] **API 文档** - 新接口有 API 文档
- [ ] **变更日志** - 重要变更记录在 CHANGELOG
- [ ] **使用示例** - 提供使用示例

## 审查流程

### 1. 准备审查

```bash
# 切换到任务分支
git checkout task/<taskId>

# 查看修改的文件
git diff main...task/<taskId> --name-only

# 查看具体修改
git diff main...task/<taskId>
```

### 2. 逐文件审查

对每个修改的文件：

1. **理解目的** - 这个文件要实现什么功能？
2. **检查逻辑** - 实现逻辑是否正确？
3. **应用清单** - 使用审查清单逐项检查
4. **记录问题** - 记录发现的问题

### 3. 生成审查报告

```markdown
# 代码审查报告

**任务**: <taskId>
**审查人**: @执法堂长老
**审查时间**: YYYY-MM-DD HH:MM

## 审查结果

### 通过项

- ✅ 代码风格规范
- ✅ 参数校验完整
- ✅ 异常处理完整

### 需要改进

#### 问题 1: 魔法值未提取

**文件**: `src/user.js:45`
**问题**: 硬编码了密码最小长度 8
**建议**: 提取为常量 `MIN_PASSWORD_LENGTH`

#### 问题 2: 缺少单元测试

**文件**: `src/user.js`
**问题**: `validateEmail` 函数没有单元测试
**建议**: 添加单元测试，覆盖正常和异常情况

## 总结

- 通过项: 15
- 需改进: 2
- 严重问题: 0

**建议**: 修复上述问题后可以合并
```

### 4. 反馈和修复

```
@阵堂长老

代码审查发现以下问题，请修复：

1. 魔法值未提取 (src/user.js:45)
2. 缺少单元测试 (src/user.js)

详见: 天玄宗/宗门任务榜/<taskId>/执法堂-代码审查报告.md
```

## 审查技巧

### 高效审查

1. **先看大局** - 先理解整体架构，再看细节
2. **关注重点** - 重点关注核心逻辑和安全问题
3. **使用工具** - 使用静态分析工具辅助
4. **限制时间** - 单次审查不超过 1 小时

### 建设性反馈

1. **具体明确** - 指出具体的文件和行号
2. **解释原因** - 说明为什么需要改进
3. **提供建议** - 给出具体的改进建议
4. **肯定优点** - 也要指出做得好的地方

### 常见问题

#### 问题 1: 过度审查

**症状**: 纠结于代码风格细节
**解决**: 使用自动化工具处理格式问题，关注逻辑和设计

#### 问题 2: 审查不足

**症状**: 只看表面，没有深入理解
**解决**: 确保理解代码意图，必要时运行代码

#### 问题 3: 主观判断

**症状**: 基于个人偏好而非标准
**解决**: 遵循团队规范，有争议时讨论

## 自动化工具

### 静态分析

```bash

npm run lint

# Java
mvn checkstyle:check
mvn pmd:check

# Python
pylint src/
mypy src/
```

### 代码格式化

```bash
# JavaScript/TypeScript
npm run format

# Java
mvn spotless:apply

# Python
black src/
```

### 测试覆盖率

```bash
# JavaScript/TypeScript
npm run test:coverage

# Java
mvn jacoco:report

# Python
pytest --cov=src
```

## 注意事项

1. **尊重开发者** - 代码审查是协作，不是批评
2. **及时反馈** - 尽快完成审查，不要拖延
3. **双向沟通** - 鼓励讨论，不要强制
4. **持续改进** - 从审查中学习，改进流程
