# 宗门新任务

## 概述

`/宗门新任务` 命令用于启动一个新的开发任务。宗主会主导需求探索、任务拆解和工作分配，确保任务目标清晰、计划完善。

## 🚨 重要提醒给宗主

**新的工作流程（先创建任务，再讨论更新）：**

1. **Phase 1: 立即创建任务（状态：新建）+ 需求探索**
   - 收到用户任务后，**第一件事**就是创建任务目录和初始文件
   - 任务状态：🆕 新建
   - 然后开始需求探索，**持续更新任务文件**
   - 任务文件是讨论的"工作台"

2. **Phase 2: 确认任务（状态：待接）**
   - 用户确认最终方案后，完善所有任务文件
   - 更新任务状态：🆕 新建 → ⏸️ 待接
   - 确保任务文件完整、自包含

3. **Phase 3: 开始执行（状态：执行中）**
   - 更新任务状态：⏸️ 待接 → 🔄 执行中
   - 调度各堂长老开始工作

**任务状态流转**：
```
🆕 新建（需求探索中，持续更新任务文件）
   ↓ 用户确认方案
⏸️ 待接（任务已确认，准备执行）
   ↓ 开始调度
🔄 执行中（各堂长老工作中）
   ↓ 测试通过
✅ 已完成（归档）
```

**HARD-GATE 检查点**：
- ✅ 收到任务后，**立即**创建任务目录和初始文件（状态：新建）
- ✅ 每次和用户确认设计点后，**立即更新**任务文件并提交
- ❌ 未经用户确认最终方案前，禁止将状态改为"待接"
- ✅ 用户确认后，完善任务文件，更新状态为"待接"
- ✅ 状态改为"待接"后，立即调度各堂开始执行

## 执行流程

### 第一阶段：创建任务并探索需求（宗主）

**🚨 第一步：立即创建任务（状态：新建）**

收到用户任务后，宗主**第一件事**就是创建任务：

```bash
# 1. 生成任务ID
TASK_ID=$(date +"%Y-%m-%d")-[任务简称]

# 2. 创建任务分支
git checkout -b task/$TASK_ID

# 3. 创建任务目录
mkdir -p 天玄宗/宗门任务榜/$TASK_ID

# 4. 创建初始任务文件（草稿版）
cat > 天玄宗/宗门任务榜/$TASK_ID/任务总览.md <<EOF
# $TASK_ID

## 任务描述
[用户原始需求]

## 当前状态
🆕 新建 - 需求探索中

## 待确认问题
- [ ] 技术栈？
- [ ] 数据库设计？
...
EOF

# 5. 创建状态文件（状态：新建）
cat > 天玄宗/宗门任务榜/$TASK_ID/.task-status.json <<EOF
{
  "taskId": "$TASK_ID",
  "status": "新建",
  "createdAt": "$(date -u +"%Y-%m-%dT%H:%M:%SZ")",
  "phase": "需求探索"
}
EOF

# 6. 提交初始版本
git add 天玄宗/宗门任务榜/$TASK_ID/
git commit -m "feat: 创建任务 $TASK_ID (状态：新建)"
```

**向用户汇报**：
```
✅ 已创建任务：$TASK_ID
📋 任务状态：🆕 新建
现在开始需求探索，我会持续更新任务文件...
```

**第二步：需求探索（持续更新任务文件）**

宗主使用 **需求探索术** 进行深入的需求分析：

1. **理解用户意图**
   - 询问关键问题澄清需求
   - 了解业务背景和使用场景
   - 确认技术约束和性能要求

2. **探索项目上下文**
   - 检查现有代码结构
   - 查看配置文件和依赖
   - 查看 git 历史了解最近变更

3. **委派藏经阁调研**（如需要）
   - 如涉及新技术或第三方平台
   - 藏经阁长老调研官方文档

4. **设计方案讨论**
   - 提出多个技术方案
   - 分析各方案优劣
   - 与用户确认最终方案

5. **🚨 重要：每次讨论后更新任务文件**
   - 每次和用户确认一个设计点后，立即更新任务文件
   - 提交更新：`git commit -m "update: 更新任务 $TASK_ID - [本次确认的内容]"`
   - **任务文件是讨论的"工作台"，必须实时同步！**

### 第二阶段：确认任务并更新状态为"待接"（宗主）

用户确认最终方案后，宗主**必须**完善任务文件并更新状态：

**步骤 1: 完善任务文件**

将讨论确认的所有内容写入任务文件，确保完整、自包含：
- 更新 `任务总览.md` - 完整的需求、方案、数据库设计、接口定义
- 创建 `执行计划.md` - 详细的执行步骤和时间估算
- 创建 `阵堂任务.md` - 完整的后端实现定义（如需要）
- 创建 `器堂任务.md` - 完整的前端实现定义（如需要）
- 创建 `执法堂验收.md` - 验收标准和测试用例

**步骤 2: 更新任务状态为"待接"**

```bash
# 更新状态文件
cat > 天玄宗/宗门任务榜/$TASK_ID/.task-status.json <<EOF
{
  "taskId": "$TASK_ID",
  "status": "待接",
  "createdAt": "[创建时间]",
  "confirmedAt": "$(date -u +"%Y-%m-%dT%H:%M:%SZ")",
  "phase": "待执行",
  "tasks": {
    "阵堂": {"status": "pending"},
    "器堂": {"status": "pending"},
    "执法堂": {"status": "pending"}
  }
}
EOF

# 提交最终版本
git add 天玄宗/宗门任务榜/$TASK_ID/
git commit -m "feat: 任务 $TASK_ID 确认完成，状态：待接"
```

**向用户汇报**：
```
✅ 任务确认完成！
📊 任务状态: 🆕 新建 → ⏸️ 待接
📝 任务文件已完善
准备开始执行...
```

### 第三阶段：开始执行任务（宗主调度各堂）

**步骤 1: 更新任务状态为"执行中"**

```bash
# 更新状态
jq '.status = "执行中" | .startedAt = "'$(date -u +"%Y-%m-%dT%H:%M:%SZ")'"' \
  天玄宗/宗门任务榜/$TASK_ID/.task-status.json > tmp.json
mv tmp.json 天玄宗/宗门任务榜/$TASK_ID/.task-status.json

git add 天玄宗/宗门任务榜/$TASK_ID/.task-status.json
git commit -m "update: 任务 $TASK_ID 开始执行"
```

**步骤 2: 调度各堂长老**

宗主**必须**使用 **任务执行法** 调度各堂：

#### 步骤 1: 生成任务 ID 和创建分支

```bash
# 生成任务ID（格式：YYYY-MM-DD-任务简称）
TASK_ID=$(date +"%Y-%m-%d")-user-login  # 示例

# 创建并切换到任务分支
git checkout -b task/$TASK_ID
```

#### 步骤 2: 创建任务目录结构

```bash
# 创建任务目录
mkdir -p 天玄宗/宗门任务榜/$TASK_ID

# 目录结构如下：
# 天玄宗/宗门任务榜/<任务ID>/
# ├── 任务总览.md          # 需求和方案说明
# ├── 执行计划.md          # 详细的分步计划
# ├── 阵堂任务.md          # 后端开发任务（如需要）
# ├── 器堂任务.md          # 前端开发任务（如需要）
# ├── 执法堂验收.md        # 测试验收标准
# └── .task-status.json    # 任务状态跟踪
```

#### 步骤 3: 编写任务文件（必须包含完整内容）

**任务总览.md** - 必须包含：
- 需求描述
- 技术方案
- 数据库设计（如有）
- API 设计（如有）
- 验收标准

**执行计划.md** - 必须包含：
- 完整的实现步骤
- 每步的具体代码
- 测试命令和预期输出
- 遵循 No Placeholders 原则

**阵堂任务.md / 器堂任务.md** - 必须包含：
- 完整的代码实现
- 具体的文件路径
- 详细的测试步骤

**执法堂验收.md** - 必须包含：
- 功能测试用例
- 质量审查标准
- 集成测试步骤

#### 步骤 4: 创建状态文件

```bash
cat > 天玄宗/宗门任务榜/$TASK_ID/.task-status.json <<EOF
{
  "taskId": "$TASK_ID",
  "status": "in_progress",
  "createdAt": "$(date -u +"%Y-%m-%dT%H:%M:%SZ")",
  "createdBy": "@宗主",
  "branch": "task/$TASK_ID",
  "description": "[任务描述]",
  "tasks": {
    "藏经阁": {"status": "pending", "blockedBy": []},
    "阵堂": {"status": "pending", "blockedBy": ["藏经阁"]},
    "器堂": {"status": "pending", "blockedBy": ["阵堂"]},
    "执法堂": {"status": "pending", "blockedBy": ["阵堂", "器堂"]}
  }
}
EOF
```

#### 步骤 5: 提交任务文件

```bash
git add 天玄宗/宗门任务榜/$TASK_ID/
git commit -m "feat: 创建任务 $TASK_ID"
```

### 第三阶段：立即开始执行（宗主）

**重要**：任务创建完成后，宗主**必须立即**开始执行，不要等待用户再次指示。

**调度透明度要求**：
- ✅ 必须明确告诉用户当前调度了哪个长老
- ✅ 必须说明是单个 Agent 执行还是多个 Agent 协作
- ✅ 必须说明执行顺序和依赖关系
- ✅ 每个长老完成后，宗主要汇报进度并调度下一个

宗主会按照依赖关系自动调度各堂执行：

**宗主的调度方式**：

#### 调度模式说明

**单个 Agent 顺序执行**（默认模式）：
- 一次只调度一个长老
- 当前长老完成后，宗主汇报进度并调度下一个
- 适合有明确依赖关系的任务

**多个 Agent 协作**（特殊情况）：
- 当多个任务可以并行时，宗主会同时调度多个长老
- 例如：阵堂和器堂可以并行开发时
- 宗主会明确说明"现在同时调度 @阵堂长老 和 @器堂长老"

## 配置选项

```bash
# 快速模式（跳过详细探索）
/宗门新任务 --quick

# 指定任务类型
/宗门新任务 --type feature    # 新功能
/宗门新任务 --type bugfix     # Bug 修复
/宗门新任务 --type refactor   # 重构

# 指定优先级
/宗门新任务 --priority high   # 高优先级
```

## 注意事项

1. **需求必须明确**
   - 宗主会充分探索需求
   - 不明确的需求会要求用户补充

2. **遵循 HARD-GATE 原则**
   - Never guess（不猜测）
   - Never use placeholders（不用占位符）
   - Never skip steps（不跳过步骤）

3. **任务隔离**
   - 每个任务独立的 Git 分支
   - 任务完成后才能合并

4. **质量保证**
   - 执法堂两阶段测试
   - 代码审查必须通过
