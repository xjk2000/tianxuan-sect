# 代码审查请求术 (Requesting Code Review)

## 技能说明

在完成开发任务后，准备并请求代码审查。提供清晰的上下文和变更说明。

## 核心原则

**让审查者能够快速理解你的变更和意图。**

## 何时使用

**在以下情况下请求审查**：
- ✅ 完成了新功能
- ✅ 修复了 bug
- ✅ 重构了代码
- ✅ 所有测试通过
- ✅ 自检完成

## 准备审查材料

### 1. 自检

```
阵堂长老: 请求审查前先自检

(使用自检心法)

- [ ] 代码符合规范
- [ ] 测试全部通过
- [ ] 没有 TODO/FIXME
- [ ] 没有调试代码
- [ ] 文档已更新
```

### 2. 整理变更

```
阵堂长老: 整理变更内容

git diff main...feature-branch

记录:
- 修改了哪些文件
- 添加了哪些功能
- 删除了什么
- 为什么这样改
```

### 3. 准备说明

```markdown
## 代码审查请求

### 变更概述
实现了用户登录功能，包括邮箱密码验证和 JWT token 生成。

### 变更文件
- `src/services/auth.service.ts` - 新增认证服务
- `src/controllers/auth.controller.ts` - 新增认证控制器
- `src/routes/auth.routes.ts` - 新增认证路由
- `tests/auth.service.test.ts` - 新增测试

### 关键变更
1. **认证服务** - 使用 bcrypt 加密密码，使用 JWT 生成 token
2. **错误处理** - 统一的错误响应格式
3. **测试覆盖** - 覆盖率 90%

### 测试结果
```bash
npm test
✅ 所有测试通过 (45/45)
覆盖率: 90%
```

### 注意事项
- JWT secret 存储在环境变量中
- Token 有效期为 24 小时
- 密码最少 8 位

### 审查重点
请重点关注：
1. 密码加密是否安全
2. JWT 配置是否合理
3. 错误处理是否完善
```

## 变更文件

### 新增文件

- `tests/auth.service.test.ts` - 测试

### 修改文件
- `src/app.ts:50-55` - 添加认证路由

### 删除文件
- 无

## 功能说明

### 1. 用户登录
- 输入: 邮箱 + 密码
- 验证: bcrypt 验证密码
- 输出: JWT token

### 2. Token 生成
- 算法: HS256
- 有效期: 24 小时
- 载荷: userId, email

## 技术决策

### 为什么使用 bcrypt？
- 行业标准
- 自动加盐
- 计算成本可调

### 为什么使用 JWT？
- 无状态
- 易于扩展
- 支持跨域

## 审查请求检查清单

提交审查前检查：

- [ ] **自检完成** - 使用自检心法
- [ ] **测试通过** - 所有测试绿色
- [ ] **文档完整** - 变更说明清晰
- [ ] **提交整洁** - 提交消息有意义
- [ ] **上下文充分** - 审查者能理解变更
- [ ] **重点明确** - 说明需要关注的地方
- [ ] **已知问题列出** - 不隐瞒问题

## 总结

好的审查请求应该：
- ✅ 提供充分的上下文
- ✅ 说明变更的原因
- ✅ 突出关键决策
- ✅ 展示测试结果
- ✅ 指出审查重点
- ✅ 列出已知问题

**让审查者的工作更轻松，审查会更快更好。**
